## Decisions

### Concurrency cap
`executionLoop` slices ready nodes: `readyNodes.slice(0, maxConcurrentNodes)` per iteration.
Remaining ready nodes are picked up on the next poll cycle. This prevents burst overload while
keeping the loop simple — no token bucket or semaphore needed.

### HeartbeatManager
One `setInterval` per node ID, stored in a `Map<nodeId, timer>`. `start()` is a no-op if a
timer already exists for that node (prevents duplicate timers on resume). `stopAll()` is called
in `markShutdownNodes()` before touching DB to guarantee no stale pulse fires after shutdown.

### recoverOrphans — null heartbeat ignored
Nodes with `last_heartbeat = null` are NOT considered orphaned. `null` means the process
crashed before the first heartbeat was written — we can't know the true start time, so we
leave them for manual admin resolution. Only nodes with a stale non-null heartbeat are marked.

### markShutdownNodes — INTERRUPTED not FAILED
Shutdown interruption is not a node failure. Marking as `INTERRUPTED` (Am.65) allows
`resumeRun()` to replay from that node's last stable handoff. `FAILED` would require admin
intervention and a manual retry. The `_interruptedRunIds` set feeds `suspendInterruptedRuns()`.

### Git branch parent chain
All feat branches were rebased so each starts from the develop commit that existed when the
branch was "opened" (simulate realistic PR workflow). feat/specs starts from main; each
subsequent feat/* starts from develop after the previous feat was merged.

### SIGTERM handler
Registered via `process.once('SIGTERM', ...)` in the factory (not in CustomExecutor) so it is
only registered once regardless of how many engine instances are created in tests. The factory
disables it automatically when `db` is injected (test context hint).
