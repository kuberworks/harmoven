## Decisions

### IntentClassifier — model tier & token cap
`ILLMClient.chat()` called with `{ model: 'fast', maxTokens: 500 }` per spec (Section 5.1).
The real tier routing (fast → GPT-4o-mini / Claude Haiku / fastest Ollama) is wired in T1.9.
The mock always returns whatever is in the response queue, making tests fully deterministic.

### requires_clarification derived field
`requires_clarification` is not part of the LLM JSON output — it is derived by the
`IntentClassifier` after parsing: `confidence < 80 → true`. This prevents an inconsistency
where the LLM could claim `requires_clarification: false` but report `confidence: 55`.
A single source of truth for the gate condition.

### requires_human_approval derived field
Same pattern in the `Planner`: `meta.confidence < 85 → requires_human_approval = true`.
Defined by the spec (Section 5.2) — UI shows a checkpoint card before DAG execution begins.

### Strict JSON parse — no silent fallback
Both agents throw a descriptive error on invalid LLM JSON rather than returning a stub
fallback. Invalid planner output must never silently create a broken DAG. Error messages
include the first 200 characters of the raw response to aid debugging.

### Types — export strategy
All shared types (`ProfileId`, `OutputType`, `Domain`, `ClassifierResult`) are exported from
`lib/agents/classifier.ts`. `PlannerHandoff` and related types are exported from
`lib/agents/planner.ts`. Downstream consumers (T1.7 Writer, T1.8 SSE routes) can import
from these files directly — no extra types barrel needed at this stage.

### MockLLMClient usage
The test fixtures use `setNextResponse()` with hand-crafted JSON strings that match the exact
schema the agents expect. Tests assert on both the returned struct and on `llm.calls` metadata
(model tier, message roles, content snippets) to catch regression on prompt construction.
