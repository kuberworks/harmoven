## 3. Architecture decisions

### 3.1 Auth model for `admin:audit`

Amendment 85 specifies `admin:audit` as an instance-level permission. Since `resolvePermissions()` requires a `project_id`, and analytics can be cross-project, I implemented the check directly on `caller.instanceRole === 'instance_admin'` — consistent with how the RBAC code handles instance admin bypass.

Project-scoped users (project admin or API key) get a filtered view restricted to their `project_id`, always anonymized (Am.85.10).

### 3.2 PDF export without Puppeteer

Puppeteer is referenced in Am.85.9 as "already available as MCP skill". Since it's not a direct npm dependency in `package.json`, the `toPdfHtml()` function returns a self-contained HTML document with `@media print` styles optimized for PDF printing. The route returns it with `Content-Type: text/html; charset=utf-8`. Browsers can "Print to PDF" natively; a Puppeteer MCP step can wrap the HTML and produce a proper PDF in a higher-level workflow.

### 3.3 Shared handler module

`lib/analytics/handler.ts` contains two shared functions:
- `buildAnalyticsResponse()` — orchestrates parallel computation of all analytics sections
- `parseAnalyticsQuery()` — validates and normalizes URL search params

Both routes (`/api/analytics` and `/api/v1/analytics`) import from the handler, avoiding duplication. The only difference is that `/api/v1/analytics` accepts API key callers routed through `assertProjectAccess()`.

### 3.4 Security

All user-facing string values in the PDF/HTML export pass through `htmlEscape()` to prevent XSS. CSV output uses RFC 4180-compliant `csvEscape()`. No raw SQL is used anywhere — all DB access goes through parameterized Prisma queries.

Date range is capped at 732 days (2 years) to prevent DoS via unbounded result sets.

## 6. API surface

### `GET /api/analytics` and `GET /api/v1/analytics`

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `from` | ISO 8601 date | Yes | UTC |
| `to` | ISO 8601 date | Yes | Must be after `from` |
| `project_id` | UUID | Only for non-admins | Required for project-scoped access |
| `granularity` | `day`\|`week`\|`month` | No | Default: `week` |
| `format` | `json`\|`csv`\|`pdf` | No | Default: `json` |

Response structure:
```typescript
{
  period:      { from, to },
  granularity: 'week',
  summary:     AnalyticsSummary,
  timeseries:  AnalyticsDataPoint[],
  by_profile:  ProfileBreakdown[],
  by_user:     UserBreakdown[],
  top_kpis:    BoardKPI[],
}
```

### `PATCH /api/runs/:runId/feedback`

Body:
```typescript
{
  user_rating?:            1 | 2 | 3 | 4 | 5,
  estimated_hours_saved?:  number,           // non-negative
  business_value_note?:    string,
}
```

Constraints: run must be `COMPLETED`. Caller needs `runs:read` in the run's project.
