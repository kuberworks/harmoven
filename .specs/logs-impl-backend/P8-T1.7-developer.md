## Decisions

### Writer — LLM tier from complexity
`low → fast`, `medium → balanced`, `high → powerful` per spec Table 5.3.
The mapping is a plain `Record<Complexity, string>` — no function, no switch.
Real model names (Haiku, Sonnet, Opus) are resolved by the LLM router in T1.9.

### Writer — streaming via onChunk
`execute()` accepts an optional `onChunk: (chunk: string) => void` callback.
When provided it uses `llm.stream()`; otherwise `llm.chat()`.
SSE wiring (T1.8) will pass a `res.write` adapter as `onChunk`. No SSE code here.

### Writer output schema
`WriterOutput` uses `execution_meta.tokens_input` / `tokens_output` (snake_case, not camelCase)
to be consistent with the DB `Node` model field names and the handoff JSON spec.
This caused the one test failure caught immediately and fixed before merge.

### Reviewer — always powerful tier
`llm.chat({ model: 'powerful' })` — no dynamic tier for Reviewer. Per spec Section 5.4:
"LLM Assignment: Always highest-capability LLM."

### Reviewer — verdict validation
Rather than silently defaulting on an unknown verdict, the Reviewer throws
`"Reviewer: invalid verdict ..."`. A corrupted verdict must never silently reach
the human gate as APPROVE.

### Reviewer system prompt — profile-specific checks
The universal checklist + all 5 profile-specific check rules (spec Table 5.4) are embedded
in the system prompt directly. No profile YAML loaded at this stage — profiles are
loaded as part of the T1.9 LLM routing work.
