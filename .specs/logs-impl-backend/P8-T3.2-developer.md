## 4. Technical decisions

### 4.1 Per-node AbortController linked to run-level signal

`_nodeAbortControllers: Map<string, AbortController>` is keyed by the node DB primary key (not `node_id`).
Each controller is created at the top of `executeNode()` and linked to the run-level signal via:

```ts
signal.addEventListener('abort', () => nodeController.abort(), { once: true })
```

This ensures:
- `cancelRun()` (which aborts the run-level signal) still aborts all nodes
- `interruptNode('n2')` only aborts n2, leaving n1 running

### 4.2 Singleton executor for API routes (`getExecutionEngine()`)

Per-node AbortControllers live in the executor's in-memory state.
For API routes to call `interruptNode()` on the **same** instance that is executing nodes,
a process-lifetime singleton is required.

Added `getExecutionEngine()` to the factory using `globalThis.__harmoven_execution_engine`
as the carrier (standard Next.js pattern). Tests continue to call `createExecutionEngine()`
directly and inject mocks — no test pollution.

### 4.3 State machine extension (INTERRUPTED → PENDING, INTERRUPTED → COMPLETED)

Am.65 specified `INTERRUPTED → RUNNING`. The gate resolution requires two more:

| Transition | Gate decision |
|-----------|---------------|
| INTERRUPTED → PENDING | `replay_from_scratch`, `resume_from_partial` |
| INTERRUPTED → COMPLETED | `accept_partial` |

These are added to `NODE_TRANSITIONS` in state-machine.ts.

### 4.4 `last_completed_node_at` — Am.64 context filtering

The spec says injections are only shown to agents that start **after** the injection was created.
The field `last_completed_node_at` is set in `executeNode()` on the success path.
The agentRunner / context-builder layer reads it (out of scope for the executor itself) to
filter `run.user_injections` to only those created after `last_completed_node_at`.

### 4.5 `resolveInterruptGate()` — `accept_partial` re-enter via `executeRun()`

After a gate decision, `executeRun()` is called to re-enter the DAG. For `accept_partial`,
the node is set to COMPLETED with `partial_output` as `handoff_out`. `executeRun()` picks up
remaining PENDING nodes (if any). If all nodes are COMPLETED/SKIPPED, the loop exits
and the run finalizes normally.

### 4.6 Streaming partial output — MVP scope

Am.65 requires `invokeAgentStreaming()` with `onChunk` callbacks. This is deferred:
the executor stores `partial_output: null` when a node is interrupted in the non-streaming
case. The gate `resume_from_partial` accepts `edited_partial` from the **request body**
(user manually provides/edits the partial in the UI), so the feature is usable even without
streaming. Full streaming support would require extending `AgentRunnerFn` — out of T3.2 scope.

### 4.7 Security

- All 5 API routes: `resolveCaller()` → `assertProjectAccess()` → `assertRunAccess()` → `resolvePermissions()` → specific permission check
- `pause/resume`: `runs:pause`
- `inject`: `runs:inject` + 2000 char limit enforced at both route and executor layers
- `interrupt`: `runs:abort`
- `gate`: `gates:approve`
- No user-supplied data is interpolated into queries — all Prisma parameterized calls
- `content` is `.trim()`-ed and length-validated before storage (injection safety)

## 5. Self-evaluation

### Strengths

- **Clean discovery phase** — avoided duplication by reading the executor in full before writing any code
- **Singleton pattern** — the `getExecutionEngine()` singleton solves a real architectural problem (per-node AbortController sharing) correctly without hacks
- **40 % error reduction** — my interface changes fixed 13 pre-existing TypeScript errors while introducing zero new ones (34 → 21)
- **20 tests, 100 % pass, 0 regressions** — all scenarios covered including edge cases
- **No streaming scope creep** — deferred correctly with clear documentation of why

### Weaknesses / Debt

- **Streaming partial output** is not implemented: `partial_output` will be null unless the agentRunner stores it externally. This is documented in §4.6 and can be addressed in T3.3.
- **`interruptNode()` fallback path**: if the per-node AbortController is not found in-memory (API call reaches a different process replica or controller was already cleaned up), the method does a direct DB update without going through `transitionNode()`. This bypasses the audit log for the state transition itself (but a separate `node_interrupted` audit entry is still written). In a multi-replica setup, a proper DB polling mechanism would be needed.
- **`resolveInterruptGate()` calls `executeRun()`** synchronously and awaits it. For long runs this means the API route handler blocks until the run completes. Ideally `executeRun()` would be fire-and-forget here — left as-is to match existing pattern from `resumeRun()`.
