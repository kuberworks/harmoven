## Implementation decisions

### D1: `auto_install: notify` as default (not `false`)
The spec says `notify | auto | manual`. The existing `orchestrator.yaml` had `auto_install: false`. Migrated to `notify` (most conservative non-disabling option). This is a breaking change to the config schema — documented in the comment block. No migration script needed since orchestrator.yaml is a project-level config file, not DB data.

### D2: Static SQL risk classification (not a full SQL parser)
The spec says "migration preview with risk warnings". Implemented keyword-based classification (DROP TABLE/COLUMN → danger; ALTER → warning; CREATE → safe). This is intentionally simple — a full parser (like libpg_query) would be overkill for the preview use case and would add a significant native dependency. The false-positive rate is acceptable: conservative classification (things may be flagged as warning when safe, never the opposite).

### D3: `docker manifest inspect` for digest verification (not `docker pull`)
Am.91.6 requires digest verification before applying an update. Used `docker manifest inspect` (read-only, doesn't pull layers) rather than `docker inspect` (requires local image). This lets the verify step run before the slow `docker pull` in the apply flow. If `manifest inspect` fails (no docker CLI / no daemon), we log a warning and continue — non-fatal since `electron-updater` verifies natively for the Electron path.

### D4: electron/auto-updater.ts scoped to Electron shell only
The file imports `electron` and `electron-updater` which are not in this project's `package.json` (Next.js app). TypeScript reports these as module-not-found errors. This is intentional and consistent with the existing codebase pattern (`lib/config-git/paths.ts` already imports from `electron` with the same pre-existing error). The Electron shell will have its own `package.json` with these dependencies when the Electron target is built.

### D5: SHA512 verification from GitHub release body
The spec (Am.91.6) describes cross-checking the downloaded electron-updater file against a SHA512 hash from the GitHub release. Implemented by parsing `SHA512: <hex>` lines from the `body` field of the GitHub Release API response. If the hash is not present in the release body, we log a warning and proceed (electron-updater already verifies the code-signing certificate natively). This is belt-and-suspenders security, not a hard gate on missing hashes.

### D6: `waitForHealth()` polls own `/api/health` after docker compose up
The spec says "Health check — auto-rollback if /api/health fails in 60s". In a Docker Compose setup, the new container needs time to start. The health poll uses 2-second intervals over 60 seconds (30 retries). The `baseUrl` comes from `process.env.AUTH_URL` — same env var already used by Better Auth. If `AUTH_URL` is not set, falls back to `localhost:3000`.

### D7: `assertInstanceAdmin()` for both update routes
Update management is instance-level (not project-scoped). Both `GET /api/updates` and `POST /api/updates/apply` enforce `assertInstanceAdmin()` from `lib/auth/rbac.ts`. Project admins can't trigger updates. This is consistent with the existing Admin Skills route pattern.

## What was NOT implemented (deferred)

- **Scheduled background polling (Docker mode)**: `orchestrator.yaml` has `check_interval_hours: 24` but there's no daemon / cron job that calls `GET /api/updates` automatically. In Docker mode, this check is user-initiated from the Admin UI (or triggered by Next.js server-side rendering on the Admin page). A proper background worker would require a server startup hook (T3.10+).
- **Maintenance window selection**: The spec mentions "schedule update for maintenance window (Sunday 3h local)". The `UpdateBanner` wizard goes straight to `[Update now]` — there's no scheduler UI / cron integration. Deferred to a future task.
- **Admin email notification after auto-install**: The spec says "Admin notified via email + UI after completion". Email integration is not yet wired.
- **`UpdateBanner` integrated into Admin layout**: The component is built but not wired into `app/api/admin/` layout or any server component. Integration requires a server component that calls `checkForUpdates()` on page load and passes the result to `UpdateBanner`.
